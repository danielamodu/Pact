Developer Guide

Pact Protocol

On-chain recurring pull payments using EIP-7702 smart account upgrades and session key delegation. No custodians. No recurring approvals. Fully autonomous.

Overview

Pact enables gas-efficient, decentralised recurring payments by upgrading a subscriber's EOA to a smart contract wallet via EIP-7702. Users sign a one-time session scope that authorises a merchant's session key to pull a fixed amount on a fixed interval — nothing more.

Key properties

  • • Non-custodial — no contract holds funds; pulls go directly subscriber → merchant
  • • Scoped authority — session key can only pull ≤ maxAmount every ≥ interval seconds
  • • Revocable — subscriber can revoke at any time on-chain
  • • Cross-chain — Arbitrum One and Base Mainnet supported

How It Works

01

Merchant creates a plan

Token, price, interval, payout address registered on PactRegistry.

02

Subscriber signs once

Browser generates an ephemeral session key. Subscriber signs a delegation scope and an EIP-7702 authorization to temporarily upgrade their EOA.

03

Keeper runs autonomously

Every 5 minutes the Pact keeper checks all stored delegations, validates intervals and balances, then submits executePull() from each session key wallet.

04

Merchant app gets notified

A signed webhook is POSTed to your server. Verify it and grant or revoke access to your service.

Session Keys

A session key is an ephemeral EOA generated in the browser at subscribe time. It is granted a narrow scope:

{
  sessionKeyAddress: "0xABC...",  // ephemeral key address
  recipient:         "0xMERCHANT...",
  maxAmount:         "9990000",    // 9.99 USDC (6 decimals)
  token:             "0xUSDC_ADDRESS",
  interval:          2592000,      // 30 days in seconds
  expiry:            1798761600,   // unix timestamp
  planId:            4
}

The subscriber signs this scope with their main wallet. The session key private key is encrypted (AES-256-GCM) and stored in the Pact keeper database. It can only pull the exact allowed amount, to the exact recipient, no more frequently than the interval.

Pact Registry

Merchants create plans on-chain. A plan defines the billing terms.

// Create a plan (merchant wallet required)
import { ethers } from "ethers";

const registry = new ethers.Contract(
  "0x9Db4207Da96c5ee738F19B54aa4D49Bc0FA64F56",
  ["function createPlan(string name, address token, uint256 price, uint256 intervalSeconds, address payoutAddress) external returns (uint256 planId)"],
  merchantSigner
);

const tx = await registry.createPlan(
  "Premium",
  USDC_ADDRESS,
  ethers.parseUnits("9.99", 6),
  2592000,  // 30 days
  merchantPayoutAddress
);
const receipt = await tx.wait();
// planId emitted in PlanCreated event

EIP-7702 Account Upgrade

EIP-7702 lets a standard EOA temporarily behave as a smart contract. When a subscriber signs a Type 4 authorization transaction delegating to SessionKeyExecutor, their EOA gains executePull() and all session key verification logic — without deploying a separate contract.

Important

When executePull() is called on the subscriber's EOA, address(this) equals the subscriber's address — not the executor contract address. All signature verification uses the subscriber's EOA as the verifying contract in the EIP-712 domain separator.

Autonomous Keeper

The Pact keeper runs every 5 minutes via Vercel Cron and cron-job.org as a fallback. For each stored delegation it checks:

  1. 1. Session key not revoked on-chain
  2. 2. Authorization not expired
  3. 3. Billing interval has elapsed since last pull
  4. 4. Subscriber has sufficient balance
  5. 5. Plan is still active

If all checks pass, the keeper funds the session key wallet with gas (if needed), then submits executePull() from that session key — satisfying the on-chain msg.sender == scope.sessionKeyAddress check.

External Integration

How to add Pact as a payment method in your app

Pact works like Stripe for on-chain subscriptions. Your app (Spotify, a SaaS tool, a content platform) doesn't touch any contracts directly — you redirect users to Pact's subscribe flow and receive webhook events when payments execute.

Integration flow

1

Create a plan on pactxbt.vercel.app — you get a planId.

2

Redirect users to the Pact subscribe URL or embed the iframe widget on your page.

3

Register a webhook — Pact POSTs a signed event to your server every time a payment executes.

4

Verify subscription status any time via the public status API — no wallet connection required on your end.

Subscription Status API

Check whether a wallet address is an active subscriber to your plan. No authentication required — subscription state is derived from on-chain data.

GET /api/v1/subscriptions/status

Query parameters

ParamRequiredDescription
planIdYesNumeric plan ID from PactRegistry
subscriberYesSubscriber's wallet address (0x...)
networkNoarbitrum (default) or base
Example request
GET /api/v1/subscriptions/status?planId=4&subscriber=0xABC...&network=arbitrum
Response — active subscriber
{
  "active": true,
  "planId": "4",
  "planName": "Premium",
  "subscriber": "0xabc...",
  "network": "arbitrum",
  "lastPull": "2026-07-15T12:00:00.000Z",
  "nextPull": "2026-08-14T12:00:00.000Z",
  "expiry": "2027-01-01T00:00:00.000Z"
}
Response — inactive
{
  "active": false,
  "planId": "4",
  "subscriber": "0xabc...",
  "network": "arbitrum",
  "reason": "No active subscription found"
}

Example — gating access in your backend:

async function isPremium(walletAddress) {
  const res = await fetch(
    `https://pactxbt.vercel.app/api/v1/subscriptions/status?planId=4&subscriber=${walletAddress}`
  );
  const data = await res.json();
  return data.active === true;
}

Webhooks

Register a webhook URL on your plan's page in the Pact dashboard. Pact will POST a signed JSON event to your URL every time the keeper successfully executes a payment.

Event: pull.executed

{
  "event":      "pull.executed",
  "planId":     "4",
  "network":    "arbitrum",
  "subscriber": "0xabc...",
  "amount":     "9990000",   // raw token units
  "txHash":     "0x123...",
  "timestamp":  "2026-07-22T18:00:00.000Z"
}

Signature verification

Every request includes two headers:

X-Pact-Signature: sha256=<HMAC-SHA256 hex digest>
X-Pact-Timestamp: 2026-07-22T18:00:00.000Z

Verify in Node.js:

import { createHmac } from "crypto";

function verifyPactWebhook(rawBody, signature, secret) {
  const expected = "sha256=" + createHmac("sha256", secret)
    .update(rawBody)
    .digest("hex");
  return expected === signature;
}

// In your webhook handler:
app.post("/pact-webhook", (req, res) => {
  const sig = req.headers["x-pact-signature"];
  if (!verifyPactWebhook(req.rawBody, sig, process.env.PACT_WEBHOOK_SECRET)) {
    return res.status(401).send("Invalid signature");
  }
  const event = JSON.parse(req.rawBody);
  // grant access to subscriber...
  res.sendStatus(200);
});

Webhook secret

Your whsec_ secret is shown once when you register the webhook. Save it immediately — Pact never displays it again. Treat it like a private key.

Embed Button

Drop an iframe into any webpage. The button shows your plan name and price and opens the Pact subscribe flow in a new tab.

<!-- Replace PLAN_ID and NETWORK -->
<iframe
  src="https://pactxbt.vercel.app/embed/PLAN_ID?network=NETWORK"
  width="320"
  height="120"
  style="border:none;border-radius:8px"
  loading="lazy"
></iframe>

No JavaScript required on your side. The embed is sandboxed and loads instantly.

Subscribe Redirect

Link or redirect your users directly to the Pact subscribe page for your plan:

https://pactxbt.vercel.app/subscription/PLAN_ID?network=NETWORK
// Example — React button
function SubscribeButton({ planId, network = "arbitrum" }) {
  const url = `https://pactxbt.vercel.app/subscription/${planId}?network=${network}`;
  return (
    <a href={url} target="_blank" rel="noopener">
      Subscribe with Pact
    </a>
  );
}
